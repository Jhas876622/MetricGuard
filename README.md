# MetricGuard — AI Metric-Consistency Auditor

**One line:** Scans an organization's scattered metric definitions (SQL, dashboards, notebooks), uses embeddings + vector similarity to detect when the *same business metric is defined differently across teams*, and uses a RAG-grounded LLM agent to recommend the canonical definition — fixing the silent reason dashboards disagree and executives stop trusting the numbers.

## Why this problem
- 67% of organizations don't fully trust their data for decisions (up from 55% the prior year).
- A 10-person data team spending ~35% of its time reconciling metrics ≈ **$525K/year**.
- Big tools (dbt Semantic Layer, Cube) attack this with heavy infra; MetricGuard is a lightweight **detector** that finds the conflicts automatically.

## Results on the demo dataset
- 12 definitions scanned · **12 in conflict (100%)** · 4 conflict groups · 6 teams affected · peak trust-risk 100/100.

## Run it
```bash
pip install -r requirements.txt          # all dependencies pinned
python src/engine.py                      # core: embeddings + conflict detection
python src/genai.py                       # GenAI: RAG retrieval + LLM agent recommendations
python src/report.py                      # writes output/results.json + output/data.js + prints KPIs
# then open output/dashboard.html in any browser (reads data.js from the same folder)
```

Run the test suite:
```bash
pytest tests/ -v
```

For real LLM output, set your API key before running:
```bash
# Windows
set ANTHROPIC_API_KEY=your_key_here
# Mac / Linux
export ANTHROPIC_API_KEY=your_key_here
python src/report.py
```
Without it, the pipeline still runs end-to-end using a transparent fallback.

## Architecture

```
data/
  metric_definitions.json   # 12 synthetic team metric definitions (the input)
  glossary.json             # 5 governed canonical definitions (the source of truth)
  concept_glossary.json     # vocabulary map — routes metric names to shared concepts

src/
  engine.py    # embed → cosine similarity → union-find grouping → conflict detection
  genai.py     # RAG retrieval (top-k) + LLM agent recommendation
  report.py    # orchestrates pipeline, writes output/results.json + output/data.js

output/
  results.json   # full machine-readable payload
  data.js        # dashboard entry-point (const MG_DATA = {...})
  dashboard.html # static executive dashboard — open in any browser

tests/
  test_engine.py  # pytest suite covering core detection logic
```

## How each interview topic maps to the code
| JD topic | Where it lives | What to say |
|---|---|---|
| SQL & analytics | `data/metric_definitions.json`, `engine.py: extract_sql_features()` | Structural SQL parsing — extracts GROUP BY, WHERE clauses, aggregation functions; detects the exact causes of "same metric, different number" |
| Embeddings | `engine.py: EmbeddingModel.encode()` | Text → 384-dim vector capturing *meaning*; MiniLM neural model, TF-IDF fallback |
| Vector DB / similarity search | `engine.py: cosine_similarity_matrix()` | Normalize to length 1 so cosine == dot product; all-pairs = one matrix multiply — what a Vector DB does at scale |
| Statistics | `trust_risk_score()`, `avg_similarity` | Quantify divergence, turn conflicts into a 0–100 risk score with documented rationale |
| ML clustering | `find_semantic_groups()` | Union-find connected components — captures transitive similarity correctly |
| RAG | `genai.py: retrieve_glossary_entry()` | Top-k retrieval → re-rank → augment prompt → generate. Grounds the LLM so it can't hallucinate the definition |
| LLM | `genai.py: call_llm()` | Claude reads messy SQL and explains what each metric actually computes; typed exception handling |
| Agentic workflow | `genai.py: resolve_conflict()` | Multi-step: analyze → retrieve → reason → recommend canonical definition |
| Schema validation | `engine.py: MetricDefinition` (Pydantic) | Every metric validated on load — catches missing fields before they cause silent None comparisons downstream |
| Data engineering | ingestion + `output/results.json` | ETL of definitions from many sources into one governed store (a mini semantic layer) |
| Product / stakeholder comms | `dashboard.html` | Trust-risk score + plain-English recommendation for a non-technical exec — filterable by team, sortable by risk |
